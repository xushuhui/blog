# 左耳听风 10 day 打卡
学习《左耳听风》第 44 讲《44 | 弹力设计篇之「幂等性设计」》。
幂等性设计是指一次请求和多次请求某一个资源有同样的副作用
系统解耦后，服务调用会存在三种状态，成功/失败/超时。超时可能是网络传输丢包，也可能是请求
到了结果没有正常返回等，对发送方来说超时原因是未知的。
针对超时有两种处理方式
-   接收方提供查询接口，发送方在超时后调用查询接口查询数据处理成功还是失败。
-   幂等性处理。接收方必须保证一次和多次请求结果一样，发送方重试
### 全局 ID
要做到幂等性交易接口，需要一个全局唯一标识标记单笔交易，如果由一个服务发放唯一标识，
会增加性能开销，如果由调用方生成，如果调用方是集群，可能会重复分配。
我们可以引入类似 UUID 的全局唯一 ID 生成算法，但是 UUID 字符串空间比较大，索引效率低，随机生成，
没有递增，无法排序。
我们可以使用 Snowflake 算法
### 处理流程
-   收到交易请求时从存储中查询一次，但是大部分请求不会重新发，所以每次查询会影响性能。
-   收到交易请求时去存储写入 ID，如果无法写入说明之前已经收到过，就不用处理
-   状态更新的场景中需要保证 where 条件包含状态和 id 字段
  ```
  update table set status = "paid" where id =: id and status = "uppaid"
  ```
## HTTP 幂等性
-   GET 方法获取资源，没有副作用产生，所以幂等
-   HEAD 仅仅是 HTTP 头信息，幂等
-   OPTIONS 获取当前 URL 支持的方法，幂等
-   DELETE 删除资源，有副作用，但是应该要满足幂等性
-   POST 创建资源，有副作用，不幂等
-   PUT 更新资源，有副作用，对同一 URL 执行多次 PUT 应该保证幂等

针对 POST 请求需要幂等的场景下的解决方式
-   在表单种隐藏一个前端生成的唯一字符串 ID，该 ID 是表单的唯一标识，
-   提交后接口把 ID 和数据保存到数据库，如果有重复提交，返回给前端不做处理，
  防止用户点击多次提交
-   更优的做法是服务端返回 302 给前端跳转到展示页面，执行 GET 请求查询刚提交的数据，
  把表单设置成过期，用户无法通过浏览器后退按扭重新提交
